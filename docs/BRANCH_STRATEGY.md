# ブランチ運用戦略 (Branch Strategy)

本プロジェクトは、基盤となるブラウザ基盤の違いにより、2つの主要なブランチを並行して運用しています。

## ブランチ一覧

| ブランチ名 | 役割 | 基盤技術 | ステータス |
| --- | --- | --- | --- |
| `main` | 旧来の本流 | Chrome / Electron (BrowserView) | メンテナンス中 |
| `kameleo-main` | **Kameleo 版の本流** | Kameleo Local API / Playwright | **アクティブ / 推奨** |
| `feature/kameleo-backend` | 履歴用 | - | 初期作業用 (Read-only) |

## 運用ルール

### 1. Kameleo 版の開発
- 今後、Kameleo 統合に関連するすべての新機能、修正、ドキュメント更新は **`kameleo-main`** を基点に行います。
- `main` ブランチとの直接的なマージ（統合）は、基盤アーキテクチャが大きく異なるため、原則として行いません。

### 2. 並行運用の目的
- `main`: 従来の BrowserView ベースの軽量なコンテナ環境を必要とするユーザー向けに維持。
- `kameleo-main`: 高度な指紋偽装やサードパーティ製プロファイル管理が必要な「Kameleo 統合モデル」向け。

### 3. 履歴の扱い
- `feature/kameleo-backend` は、Kameleo 統合に向けた初期の設計・実装フェーズの履歴を確認するための記録用として保持します。

## 注意事項
- 開発者は常に現在のカレントブランチを確認し、Kameleo 版の作業をしている場合は必ず `kameleo-main` (またはそこから派生した feature ブランチ) であることを確認してください。
- 各ブランチで作成される DB スキーマやデータ構造（特に `src/main/db.ts`）に差異があるため、同じデータパス (`userData`) で交互に起動すると不整合が発生する可能性があることに注意してください。
